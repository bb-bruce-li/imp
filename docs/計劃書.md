# imp — 認知建模層計劃書

## 讓 AI 理解你怎麼想，而不只是記住你說了什麼

> Mem0 記住你說過的話。imp 理解你的思維方式。

---

## 一、產品概述

### 1.1 一句話定義

imp 是一個介於人類與 AI 之間的認知建模層。任何 AI 工具都可以透過 imp 理解使用者的專業程度、思維模式、溝通風格和學習邊界，從而自動調整輸出方式。

### 1.2 核心問題

現有的 AI 記憶系統（Mem0、Letta）儲存的是**事實**——「使用者喜歡 Python」、「使用者在 Acme 公司工作」。

但真正讓溝通有效的不是事實，是**認知模型**：

- CEO 不在乎實作細節，在乎的是時程和成本效益
- 工程師不在乎利潤，在乎的是把產品做好
- 資料庫管理員在乎的是安全性和正規化
- 同樣是工程師，每個人的程度和理解力都不一樣

**如果你不知道 2×2，我就用 2+2 跟你說。** 同樣的答案（4），不同的編碼方式，匹配使用者能理解的程度。

### 1.3 核心洞察：近側發展區（ZPD）

每個使用者對每個概念都處於三個區域之一：

| 區域 | 定義 | AI 行為 |
|------|------|---------|
| **已掌握區** | 使用者熟練使用，不需解釋 | 直接引用，溝通壓縮 |
| **ZPD 邊緣** | 使用者正在學習，能跟上但還不獨立 | 使用概念但附帶輕量引導 |
| **超出區** | 使用者目前無法理解 | 避免使用或翻譯成已知概念 |

ZPD 邊緣會隨時間移動。當系統偵測到使用者已經能獨立使用某個概念，該概念就從「ZPD 邊緣」畢業到「已掌握」。溝通自動壓縮，對話變得更簡潔但更有意義。

這就是真正的協作——像一個與你共事多年的同事，知道什麼該詳細說，什麼可以一筆帶過。

### 2.1 競品分析

**步驟三：無感適應（永久）**

從此每次對話自動優化認知模型。使用者永遠不需要再想到 imp。

### 範例 2：寫程式碼——同一個需求，三個等級

**需求：**「幫我寫一個函數，從資料庫撈使用者的最近訂單」

---

**使用者 C：初學者 Emily（學過基礎 Python，正在學 SQL）**
```json
{
  "code_level": "beginner",
  "known_concepts": ["for loop", "if/else", "list", "dict"],
  "zpd_edge": ["SQL SELECT", "database connection", "try/except"],
  "avoid_concepts": ["ORM", "connection pooling", "async"],
  "explanation_style": "full_reasoning"
}
```

**AI 回應：**
```python
import psycopg2

def get_recent_orders(user_id):
    # 第一步：連接到資料庫
    # 這就像打電話一樣——你需要知道地址（host）、
    # 資料庫名稱、帳號和密碼
    connection = psycopg2.connect(
        host="localhost",
        database="myshop",
        user="admin",
        password="secret"
    )

    # 第二步：建立一個「游標」
    # 游標就像一個幫你執行 SQL 指令的助手
    cursor = connection.cursor()

    # 第三步：寫 SQL 查詢
    # SELECT = 我要哪些欄位
    # FROM = 從哪個表
    # WHERE = 篩選條件
    # ORDER BY = 排序方式
    # LIMIT = 最多回傳幾筆
    sql = """
        SELECT order_id, product_name, total_price, order_date
        FROM orders
        WHERE user_id = %s
        ORDER BY order_date DESC
        LIMIT 10
    """

    # 第四步：執行查詢
    # %s 會被替換成 user_id 的值
    # 這樣做比直接把值塞進字串安全（防止 SQL injection）
    cursor.execute(sql, (user_id,))

    # 第五步：拿到結果
    orders = cursor.fetchall()

    # 第六步：記得關閉連線（就像掛電話）
    cursor.close()
    connection.close()

    return orders
```

> 這個函數做了幾件事：連上資料庫、查詢指定使用者最近的 10 筆訂單、然後回傳結果。`%s` 是佔位符，用來安全地插入變數，避免 SQL injection 攻擊。下一步你可以學的是用 `try/except` 來處理連線失敗的情況。

---

**使用者 D：中階工程師 Kevin（熟 Python，會 SQL，正在學 ORM）**
```json
{
  "code_level": "intermediate",
  "known_concepts": ["SQL", "psycopg2", "context manager", "type hints"],
  "zpd_edge": ["SQLAlchemy", "Pydantic", "repository pattern"],
  "avoid_concepts": ["async SQLAlchemy", "connection pooling tuning"],
  "explanation_style": "brief_context"
}
```

**AI 回應：**
```python
from sqlalchemy import select
from sqlalchemy.orm import Session
from models import Order
from datetime import datetime
from pydantic import BaseModel

class OrderResponse(BaseModel):
    order_id: int
    product_name: str
    total_price: float
    order_date: datetime

def get_recent_orders(session: Session, user_id: int, limit: int = 10) -> list[OrderResponse]:
    stmt = (
        select(Order)
        .where(Order.user_id == user_id)
        .order_by(Order.order_date.desc())
        .limit(limit)
    )
    orders = session.scalars(stmt).all()
    return [OrderResponse.model_validate(o) for o in orders]
```

> 用 SQLAlchemy 的 `select()` 語法取代 raw SQL，讓查詢有型別安全。Pydantic BaseModel 做回傳值驗證——如果之後要接 API endpoint，這個 model 可以直接用。Session 從外面傳入而不是在函數裡建立，這是 repository pattern 的基本概念。

**AI 回應：**

---

**使用者 G：後端工程師 Tom**
```json
{
  "abstraction_level": "implementation",
  "known_concepts": ["SQL", "indexing", "connection pool", "N+1 problem"],
  "zpd_edge": ["query plan analysis", "pg_stat_statements"],
  "code_level": "intermediate",
  "framing": "root-cause"
}
```

**AI 回應：**
> 大概率是 N+1 查詢或缺少索引。先跑這個看看哪些查詢最慢：
>
> ```sql
> SELECT query, mean_exec_time, calls
> FROM pg_stat_statements
> ORDER BY mean_exec_time DESC
> LIMIT 10;
> ```
>
> 如果是某個 SELECT 在 `orders` 表上跑全表掃描，加一個索引：
>
> ```sql
> CREATE INDEX idx_orders_user_date ON orders(user_id, order_date DESC);
> ```
>
> 另外 check 一下連線池設定。如果用 SQLAlchemy，看看 `pool_size` 是不是太小了——尖峰時段連線可能排隊等待。想看一下怎麼用 `EXPLAIN ANALYZE` 分析查詢計畫嗎？

注意：AI 主動提出教 Tom 使用 `EXPLAIN ANALYZE`——因為這在他的 ZPD 邊緣。不是隨便提的，是因為模型知道他正在學這個。

---

**使用者 H：DBA Alex**
```json
{
  "abstraction_level": "implementation",
  "known_concepts": ["EXPLAIN ANALYZE", "pg_stat_statements", "vacuum", "index bloat", "connection pooling", "pgbouncer"],
  "zpd_edge": ["partitioning strategy"],
  "code_level": "expert",
  "framing": "performance-metrics"
}
```

**AI 回應：**
> `pg_stat_statements` 顯示 `get_orders_by_user` 的 mean_exec_time 從上週的 12ms 跳到 4800ms。查詢計畫從 Index Scan 變成 Seq Scan——大概率是 `idx_orders_user_date` 的 bloat 比例過高或統計資訊過期。先跑 `ANALYZE orders;`，如果沒改善就 `REINDEX CONCURRENTLY`。另外 `orders` 表已經 180GB 了，考慮按 `order_date` 做 range partition，冷資料搬到獨立 tablespace。

### 範例 4：ZPD 隨時間演進

**使用者 I：Kevin 的成長過程（同一個人，不同時間點）**

---

**第 1 週——Kevin 問：**「什麼是 ORM？」

imp 觀察到：`zpd_signal: ORM 是 beyond 區`

**AI 回應（教學模式）：**
> ORM 是 Object-Relational Mapping，簡單說就是讓你用 Python 物件來操作資料庫，而不用寫 SQL。比如你現在寫 `cursor.execute("SELECT * FROM users WHERE id = 1")`，用 ORM 就變成 `session.get(User, 1)`——效果一樣，但你操作的是 Python 物件而不是 SQL 字串...

---

**第 3 週——Kevin 問：**「SQLAlchemy 的 relationship 怎麼設定 lazy loading？」

imp 觀察到：Kevin 已經會自己用 SQLAlchemy 建 model，問的問題越來越具體。
`ORM 從 beyond → zpd_edge`
`SQL 從 zpd_edge → mastered`

**AI 回應（引導模式）：**
> 在 `relationship()` 裡設 `lazy="select"` 是預設行為——第一次存取時才查。如果你知道一定會用到關聯資料，`lazy="joined"` 會在主查詢就 JOIN 進來，避免 N+1。你的 case 是清單頁面，建議用 `selectinload()` 在查詢時明確指定...

注意：不再解釋什麼是 ORM，不再解釋什麼是 SQL。直接切入他問的具體問題。

---

**第 8 週——Kevin 問：**「在高併發場景下，async session 和 sync session 的 connection pool 行為有什麼差異？」

imp 觀察到：Kevin 能獨立使用 SQLAlchemy、理解 lazy loading 策略、開始探索 async。
`SQLAlchemy 從 zpd_edge → mastered`
`async patterns 進入 zpd_edge`

**AI 回應（同事模式）：**
> async `create_async_engine` 底層用 `asyncpg`，pool 管理和 sync 的 `psycopg2` pool 是獨立的。主要差異：async pool 在 `await session.execute()` 時釋放 greenlet 但保持連線，sync 是阻塞等待。高併發下 async 的優勢不在 DB 查詢速度（那是 I/O bound），而是同一個 process 能處理更多並行請求。pool size 建議 `min(32, cpu_count * 2 + 4)`。

8 週內，溝通從「教學」壓縮到「同事對話」。使用者感覺 AI 越來越懂他——因為它真的越來越懂。

### 範例 6：同一個錯誤訊息，不同使用者

**錯誤：**`TypeError: Cannot read properties of undefined (reading 'map')`

---

**使用者 M：React 初學者**
```json
{
  "known_concepts": ["HTML", "CSS", "basic JavaScript"],
  "zpd_edge": ["React state", "props", "array methods"],
  "code_level": "beginner"
}
```

**AI 回應：**
> 這個錯誤是說你嘗試對一個 `undefined` 值呼叫 `.map()`。最常見的原因是：你的資料還沒從 API 載入，但 React 已經在嘗試渲染了。
>
> 簡單修法——在 map 前加一個檢查：
> ```jsx
> {/* 如果 items 存在，就渲染列表；否則顯示「載入中」 */}
> {items ? items.map(item => <li>{item.name}</li>) : <p>載入中...</p>}
> ```
>
> 為什麼會這樣？React 的 state 一開始是 `undefined`，API 回應到了之後才會有值。在這中間的空檔，`.map()` 就會噴錯。你也可以用 `useState([])` 給一個空陣列作為初始值，這樣就不需要額外檢查了。

---

**使用者 N：資深 React 開發者**
```json

  "zpd_edge": [],
  "code_level": "expert"
}
```

**AI 回應：**
> Race condition——組件在 data fetch 完成前 render。如果你用 React Query，`data` 在 `isLoading` 時是 `undefined`。加個 early return 或 `data?.map()`，或者把初始值設成空陣列。

## 七、技術實作

### 7.1 信號提取管線

每個對話回合，通過 Claude API 提取隱式認知信號：

| 信號類型 | 偵測方式 | 範例 |
|----------|----------|------|
| **詞彙程度** | 使用者正確使用的最高階概念 | 隨口提到「eventual consistency」→ 分散式系統進階 |
| **抽象層級** | 使用者思考的高度 | 「讓它快一點」vs「加個 B-tree 索引」 |
| **提問模式** | 先問 why 還是 how？探索問題還是直跳方案？ | 探索型 vs 解決型 |
| **糾正行為** | 糾正 AI 錯誤時的精確度 | 模糊「那不對」vs 精確「join 順序反了」 |
| **決策模式** | 面對權衡時優化什麼 | 速度 vs 正確性、成本 vs 可靠性 |
| **信任信號** | 驗證、質疑還是接受 AI 輸出？ | 每次都要看 source vs 直接採用 |
| **ZPD 信號** | 正在學但還不熟練的概念 | 問了好問題但不會自己用 |

### 7.2 認知模型更新規則

**晉升規則：** 連續 5+ 個正面信號且最近 3 個信號都指向更高等級 → 晉升。ZPD 概念畢業到已掌握區。

**降級規則：** 軟降級（3+ 連續負面信號 → 降低信心分數）。硬降級（5+ 負面信號 → 實際降級）。

**衝突解決：** 使用者宣稱懂但表現出差距 → 不改等級，記錄「宣稱 vs 展現」差異，降低信心分數。

**時間衰減：** 舊信號逐漸失去權重。一個半年前的正面信號 < 昨天的正面信號。
